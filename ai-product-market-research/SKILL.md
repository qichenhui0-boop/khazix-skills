# AI时代产品及竞品市场研究

## 核心方法论：横纵分析法

本Skill使用「横纵分析法」（Horizontal-Vertical Analysis）作为核心研究框架。该方法融合语言学中的历时-共时分析、社会科学中的纵向-横截面研究设计、商学院案例研究法和竞争战略分析，形成一个完整的产品市场研究体系。

**两个分析轴**：

- **纵轴（Diachronic / 历时分析）**：追溯产品品类的历史演进。不是简单列时间线，而是识别每一次范式跃迁的驱动力、标志性事件和不可逆变化。核心问题："这个品类是怎么一步步走到今天的？每一步的驱动力是什么？"
- **横轴（Synchronic / 共时分析）**：在当前时间切面上对所有竞品做多维度对比。不是简单的功能对比表，而是揭示每个产品背后的战略选择和生态位差异。核心问题："此刻，不同玩家做了哪些不同的选择？为什么？"
- **交汇洞察（Cross-Axis Synthesis）**：在纵横交汇处提炼出只有同时看到两个维度才能发现的深层洞察。这是本方法论最有价值的部分。

## 研究工作流（10个阶段）

> **效率优化原则**：阶段2-5应尽可能**并行执行**。使用Task工具同时启动4个子Agent：纵向演进研究、国际竞品研究、国内竞品研究、行业趋势研究。不要串行等待——并行研究可将总耗时从60分钟缩短至20分钟。

### 阶段1：问题定框与框架设计
1. 明确研究问题（如"AI时代下BI产品的新形态"、"智能客服行业的演进路径"等）
2. 确定纵轴范围（追溯到多早？分几个阶段？）
3. 确定横轴维度（对比哪些产品？用什么维度？）
4. 设计报告骨架结构
5. **用AskUserQuestion确认**三个关键问题：
   - 研究侧重（全面分析 / 重点在产品升级建议 / 重点在行业趋势）
   - 竞品范围（国内外全覆盖 / 聚焦国内 / 聚焦AI-Native新势力）
   - 输出格式（HTML报告+GitHub Pages公开网页 / HTML+钉钉文档 / 全部格式）

### 阶段2-5：并行研究冲刺（★核心阶段，必须并行）

同时启动4个Task子Agent，每个负责一个研究维度：

**子Agent A：纵向演进研究**
1. 追溯品类的"起源故事"——找到概念诞生的最早时刻（从"需求诞生"而非"产品诞生"开始）
2. 识别3-5个关键时代划分（每个时代需要有明确的标志性事件和范式特征）
3. 分析每个时代转换的驱动力（技术突破？市场变化？用户需求演进？）
4. 用叙事驱动而非列表驱动——每个阶段都应该是一个有起承转合的故事
5. 中国市场如有独立发展脉络，单独设一节"平行宇宙"分析
6. 使用WebSearch搜索最新数据，确保时间节点准确

**子Agent B：国际竞品深度研究**
1. 按产品基因分类（传统转型/云原生/AI-Native/数据平台延伸等）
2. 每个产品研究维度：定位、核心功能、AI能力水平、技术架构、商业模式、定价、融资/估值、企业级能力、最新动态
3. 超越功能层面，理解产品哲学——它解决的根本问题是什么？
4. 评估企业就绪度（安全、合规、权限、审计）
5. 特别关注2025-2026年的最新产品更新和融资事件

**子Agent C：国内竞品深度研究**
1. 覆盖主要国内厂商（大厂生态产品+独立SaaS+AI-Native创业公司）
2. 重点分析与大厂生态的协同关系和差异化战略
3. 如有特定目标产品（如Quick Service、Quick BI等），做详细产品分析
4. 分析出海能力和国际化布局
5. 使用WebSearch获取IDC等机构的中国市场份额数据

**子Agent D：行业趋势与数据研究**
1. Gartner/IDC/Forrester等权威机构最新报告和预测
2. 行业市场规模和增长率数据（全球+中国）
3. 新兴技术趋势和应用模式
4. ROI数据和企业采用案例
5. TO B企业的特殊需求和采购逻辑

**信源增强**（在4个子Agent返回结果后执行）：
1. 整合用户提供的文章和参考资料（如有）
2. 从每篇文章中提取关键概念，验证/挑战/丰富已有框架
3. 对新概念做归类——它属于纵轴的哪个阶段？它改变了横轴的哪个维度？

### 阶段6：横纵交汇洞察与总体演进趋势总结
1. 识别只有同时看纵横两轴才能发现的模式
2. 提炼3-5个关键洞察（Key Insights），每个洞察都应该包含一个非显而易见的判断
3. 发展预测性框架——基于历史规律推演未来趋势
4. 形成"一句话定义"——用最精炼的语言概括整个研究的核心发现
5. **总体演进趋势总结**（详见[methodology.md](methodology.md)的"总体演进趋势总结方法论"章节）：
   - 从纵轴提炼1-2条**贯穿全历史的底层规律**（如"谁控制理解层，谁就控制价值"）
   - 从横纵交叉识别1-2个**正在发生的结构性拐点**（如"前后台分离"）
   - 归纳**2-4条并行演进路径**及各自的时间窗口和终局推演
   - 识别**决定市场终局的根本变量**（如TO B采购逻辑）——技术趋势只是表层，商业模式才是终局
   - 构建**多层竞争全景**——表面之争（交互形式）→深层之争（基础设施）→根本之争（商业模式），三层需同时分析
6. **市场格局判断**（详见[methodology.md](methodology.md)的"市场终局判断方法"章节）：
   - 当前市场处于什么阶段？（混战期/收敛期/稳定期？）
   - 哪些竞争维度已经commodity化？哪些是新的差异化战场？
   - 不同路径之间是互斥还是并行？赢家通吃还是多极共存？
   - 连接技术趋势与商业现实——"能做到"和"有人为此付费"是两回事

### 阶段7：TO B企业需求分析与产品升级框架（可选，针对TO B产品）
如果研究对象是TO B产品，必须深入分析企业客户的购买逻辑和需求结构。

> **领域适配指南**：TO B三维信任模型适用于所有TO B品类，但"企业级能力三支柱"需根据品类调整。
> - BI/数据产品：数据集成准备 × 数据可视化与分析 × 数据共享与协作
> - 客服/服务产品：渠道接入与路由 × AI理解与解决 × 服务数据洞察与闭环
> - CRM/营销产品：客户数据管理 × 智能触达与转化 × 效果归因与优化
> - 通用SaaS：核心功能自动化 × 协同与集成 × 数据与安全

**7a. TO B采购逻辑分析**
1. 识别目标产品的企业客户画像——谁买？怎么买？谁决策？
2. 分析购买决策的三个核心驱动力——**信任、确定性、ROI量化**（详见[methodology.md](methodology.md)的"TO B产品升级分析框架"章节）
3. 对每个竞品在三个核心驱动力维度上做能力评估
4. 识别目标产品在三个维度上的差距与机会

**7b. 企业级能力三支柱分析**
1. 根据品类确定三大能力支柱（参见阶段7开头的领域适配指南），系统分析每个支柱
2. 每个支柱：竞品的AI创新→目标产品的差距→具体演进建议
3. 三个支柱的共同方向要形成统一判断

**7c. 围绕TO B核心关切的产品升级设计**
1. **信任升级路径**：从传统安全到AI可信——推理链路透明、Agent级审计、权限继承、可信度标签
2. **确定性升级路径**：从口径统一到全链路确定性——语义消歧主动化、确定性分区设计、技术路线优势传达
3. **ROI量化升级路径**：从提供洞察到证明价值——使用价值仪表板、洞察→行动闭环追踪、价值计费探索、行业ROI基准
4. 构建三维飞轮：信任→使用频率↑→价值积累↑→ROI更清晰→续费信心↑→信任↑

### 阶段8：战略展望（可选，针对特定产品）
如果用户需要对某个特定产品做演进路径分析：
1. 结构性优势分析——不是列功能，而是分析"生态位"（为什么它占据了这个位置？这个位置的价值在AI时代是增加还是减少？）
2. 结构性挑战分析——优势的反面通常就是最大的威胁
3. 多阶段演进路径设计（通常3阶段，每阶段12个月）
4. 终局形态构想——用一个具象化的比喻来描述（如"冰山模型"）
5. 关键战略抉择点——列出必须做出的二选一决策及建议
6. **结合TO B升级框架**（如已执行阶段7）：在演进路径的每个阶段中，明确信任、确定性、ROI量化三个维度的具体升级动作

### 阶段9：视觉增强
1. 为每个主要章节生成配图（使用ImageGen）
2. 配图风格：概念性插画，低饱和度，蓝紫色调，科技感
3. 关键概念需要有可视化图解（如演进时间线、竞争矩阵、冰山模型）
4. 图片存放在outputs/images/目录

### 阶段10：多格式输出
1. **HTML报告**（主格式）——富格式、支持目录导航、关键洞察高亮、图文混排。详见[templates.md](templates.md)
2. **GitHub Pages公开网页**（公开分享格式）——将HTML报告+配图部署到GitHub Pages，生成公开可访问的网页链接。部署步骤：
   - 在outputs目录准备部署文件夹：`index.html`（报告主文件）+ `images/`（配图目录）
   - 使用git初始化本地仓库，提交所有文件
   - 通过GitHub API或浏览器创建public仓库（命名建议：`ai-[品类]-research-report`）
   - `git push`推送到GitHub仓库
   - 在仓库Settings → Pages中启用GitHub Pages（Source: Deploy from a branch, Branch: main, Folder: / (root)）
   - 等待1-2分钟部署完成，最终链接格式：`https://[username].github.io/[repo-name]/`
   - 注意：需要系统上有git且已配置Git Credential Manager
3. **钉钉文档**（团队内部共享格式）——将报告核心内容转为Markdown格式，通过钉钉文档MCP工具（`mcp__钉钉文档__create_document`）创建可分享的在线文档。转换要点：
   - Markdown中换行必须使用真实换行符（不能用`\n`转义），钉钉文档不支持HTML标签
   - 简化格式：保留表格、引用（`>`）、标题层级（`#`/`##`/`###`），去掉HTML样式类
   - 图片无法内嵌钉钉Markdown，用文字描述或省略配图
   - 表格使用标准Markdown表格语法（`|`分隔），确保对齐行（`|---|---|`）完整
   - 长报告建议精简为核心内容（控制在8000字内），避免超出钉钉文档API限制
4. **PPT版本**（衍生）——使用pptx skill从HTML报告生成，确保内容与网页版一致
5. **视频脚本**（可选）——适配特定账号风格，如"Agent早知道"

> **输出优先级**：默认同时生成HTML报告+GitHub Pages公开网页。HTML用于本地完整阅读体验，GitHub Pages用于公开分享和传播。如需团队内部协作，追加生成钉钉文档。如用户有特定格式偏好，按用户指定执行。

## 写作风格原则

详见[methodology.md](methodology.md)中的"写作风格"章节。核心要点：

1. **叙事驱动**：用讲故事的方式展开分析，不是堆砌事实和要点列表。每个分析段落应有起承转合
2. **观点鲜明**：不做"两方面都有道理"的骑墙分析。做出判断，给出理由
3. **数据支撑**：每个关键判断都需要数据或案例支撑。引用学术基准（如Spider 2.0, BIRD）、行业报告（Gartner, IDC）、产品官方数据
4. **比喻具象**：用生动比喻把抽象概念变具体（如"LLM不是在已有的路上提速了，它是修了一条新路"）
5. **非显而易见的洞察**：优先呈现读者想不到的判断。用"so what?"测试每个段落——如果读者的反应是"我知道"，那就不值得写
6. **信源充分**：报告末尾需要完整的信息来源清单，通常30-60条

## 分析工具箱

详见[methodology.md](methodology.md)。

### 通用分析工具（适用于所有品类）

- **产品基因分类法**：Traditional-transform / Cloud-Native / AI-Native / Data-Platform-Extend —— 按产品的"出生基因"而非规模或地域分类
- **多层竞争全景分析**：表面是前台交互形式之争→深层是后台基础设施之争→根本是商业模式之争。分析任何市场都需要识别这三层，只赢一层不够
- **底层规律提炼法**：从纵轴全历史中寻找贯穿始终的不变量——通常表现为"谁控制X，谁就控制Y"的结构。用于锚定趋势判断，避免被短期热点干扰
- **结构性拐点识别法**：从横轴竞争格局中寻找正在发生的不可逆结构变化——通常表现为"A和B正在分离/融合"。拐点比趋势更有预测价值
- **并行路径推演法**：避免线性预测（"未来一定是X"），而是识别2-4条并行路径，分析各自的时间窗口、适用场景和可能终局
- **市场终局三问法**：①当前commodity化的能力是什么？②新的差异化战场在哪里？③最终决定胜负的变量是技术、生态还是商业模式？
- **冰山模型**：水面之上（用户可见的交互形态）vs 水面之下（AI和系统消费的引擎层）
- **TO B三维信任模型**：信任（AI可信体系）× 确定性（全链路确定性保障）× ROI量化（可衡量价值链）——三者缺一不可
- **信任→确定性→ROI飞轮**：信任让企业敢用→确定性让企业信赖结果→ROI量化让企业持续付费→续费增强信任
- **确定性分区设计**：区分确定性操作（标准查询、自动执行）和推理性分析（归因、预测、建议），在产品层面显性标注不确定性边界
- **洞察→行动闭环评估**：产品价值不在洞察本身，在于洞察能否驱动业务行动
- **Outcome-based定价评估**：AI越强→人用得越少→Per-Seat模式越难维持。评估产品是否具备向价值计费/结果计费转型的条件

### 领域特化工具（根据研究品类选用）

**BI/数据分析品类专用：**
- **ChatBI vs Agentic BI**判断框架：ChatBI = 帮人查数据（Q&A）; Agentic BI = 帮人建数据基础设施（自主规划+多步执行+自我修正）
- **Headless BI评估**：前台交互与后台语义引擎是否解耦？语义层能否被外部Agent调用？
- **语义层成熟度**：无语义层 → 内嵌语义层 → 独立语义层 → 开放语义基础设施
- **NL2SQL精度鸿沟**：学术基准精度（Spider 1.0: 85%+）vs 真实企业环境精度（Spider 2.0: ~10%）

**客服/服务品类专用：**
- **Chatbot vs AI Agent**判断框架：Chatbot = 基于预设意图的FAQ应答; AI Agent = 自主推理+多步骤解决+Action执行（调用API退款/修改订单等）
- **解决率阶梯模型**：0→60%（标准FAQ）→60→80%（深度系统集成）→80→95%（边缘场景，边际成本急剧上升）——解决率曲线非线性
- **多模型编排成熟度**：单一LLM → 大+小模型双驱动 → 任务特化多模型协同（Constellation of Models）→ 自动故障转移+模型升级透明
- **服务价值链评估**：被动响应 → 主动服务 → 服务驱动营收 → 服务数据反哺产品改进

**CRM/营销品类专用：**
- **CDP成熟度评估**：数据收集 → 统一画像 → 智能分群 → AI驱动的自动化触达 → Agent自主营销
- **营销→服务→营销闭环**：是否实现从获客到服务到复购的完整客户旅程闭环
