# 瓴羊 Quick BI & Dataphin 公众号文章撰写 Skill

## 核心写作原则（全文必须贯穿）

**"问题先行，产品后置；用技术深度建立信任，用场景表达降低门槛"**

三个底层逻辑：

1. **Product-Led Content**：产品是解决问题的主角，自然编织进叙事，不是结尾加 CTA。"Don't lead with features. Tell your readers what problem you're solving before you explain how the tech works."
2. **价值前置，产品后置**：前 60% 纯教育内容（读者不用产品也能获得价值），后 40% 自然过渡到产品价值。
3. **"从X到Y"对比思维**：所有价值表达用"从旧状态到新状态"句式，如"从'写代码'到'写命令'"、"从'跨天开发'到'秒级开通'"。

## 写作工作流

收到用户的初始内容/素材后，严格按以下步骤执行：

### Step 1：分析输入，确认关键信息

使用 AskUserQuestion 确认以下信息（如果用户输入中已明确则跳过对应项）：

**必须确认的信息：**

| 信息项 | 判断方式 | 选项 |
|--------|----------|------|
| **文章类型** | 根据素材主题判断 | 产品功能发布文 / 深度客户案例文 / 方法论趋势洞察文 / 技术架构解析文 / 产品对比选型指南文 |
| **目标产品** | 根据内容关键词判断 | Quick BI / Dataphin / 两者结合 |
| **目标受众** | 影响语言深度和平衡模式选择 | 业务决策者 / 技术决策者 / 数据分析师 / 开发者 / 综合受众 |
| **发布平台** | 影响写法风格（默认公众号） | 公众号 / 阿里云技术社区 / InfoQ或36氪 / 知乎 / 官网产品页 |


**文章类型判断依据：**

| 类型 | 判断依据 |
|------|----------|
| **产品功能发布文** | 新产品/新功能发布、开发者工具推广、技术能力开放 |
| **深度客户案例文** | 有具体客户名称、实施效果数据、行业案例 |
| **方法论/趋势洞察文** | 行业趋势、方法论输出、战略层内容 |
| **技术架构解析文** | 技术方案、架构设计、性能优化 |
| **产品对比/选型指南文** | 产品比较、技术选型、评估维度 |

### Step 2：选择内容角度和平衡模式

根据目标产品和受众，选定内容角度和"技术-业务"平衡模式。

**Quick BI 核心角度**：智能分析架构演进、语义层设计实践、性能优化方法论、数据民主化落地、CLI工具应用、Gartner评估解读、Skill化（AI Agent集成）

**Dataphin 核心角度**：数据资产化方法论、数据治理工程化、全域数据融合架构、合规与效率平衡、OneModel方法论、非结构化数据全生命周期管理

**五种平衡模式（根据受众选择主导模式）：**

| 模式 | 策略 | 适用场景 |
|------|------|----------|
| **权威降维型** | Gartner等第三方背书替代自我论证 | 品牌建设、决策层说服（Quick BI擅长） |
| **方法论包装型** | 产品能力上升为行业方法论 | 行业影响力、架构师受众（Dataphin擅长） |
| **场景量化型** | 前后对比表格+百分比数据轰炸 | 客户案例、ROI论证 |
| **学术纵深型** | 技术演进史建立学术权威 | 技术社区、开发者受众 |
| **性能数据型** | 每个功能点配具体性能指标 | 技术选型、工程师说服 |

### Step 3：按模板结构生成文章

读取 [templates.md](templates.md) 获取对应类型的详细结构模板，严格按模板组织内容。

**不同平台的写法差异（如非公众号需调整）：**

| 平台 | 风格 | 标题 | 深度 | 配图 |
|------|------|------|------|------|
| **公众号** | 故事化、情绪驱动、短段落 | 吸引力优先，数字+悬念 | 中等，重场景效果 | 每500-800字一图 |
| **阿里云/技术社区** | 技术深度、代码示例、架构图 | 直接明了，含技术关键词 | 高，重原理实现 | 架构图、代码块、性能图表 |
| **InfoQ/36氪** | 行业视角、趋势分析 | 趋势+数据+判断 | 中高，重洞察方法论 | 趋势图、对比表格 |
| **知乎** | 问答式、干货优先 | 问题式标题，激发好奇 | 高，重论证逻辑 | 适量配图，重内容 |
| **官网产品页** | 简洁、价值导向 | 价值主张一句话 | 低，重转化 | 产品截图、演示GIF |

### Step 4：应用深入浅出表达技巧

读取 [reference.md](reference.md) 获取完整表达技巧库，在写作中必须运用：

1. **三层递进法**：类比建立认知 -> 场景化表达 -> 价值导向
2. **技术概念翻译**：所有技术术语必须同时提供业务语言和场景化描述
3. **"从X到Y"对比句式**：每篇文章至少使用3-5处

### Step 5：文章润色与自检

完成初稿后，逐项检查以下所有要点（不合格的项目需修改后再输出）：

**内容质量：**
- 开篇是否有具体痛点（而非"随着数字化转型的深入"这类空洞开场）
- 是否有"从X到Y"的对比句式（至少3处）
- 场景示例是否覆盖不同角色
- 每个场景是否完成：痛点 -> 能力 -> 方案 -> 效果 的完整链条
- 命令/操作是否真实可用（不虚构）
- 产品融入是否自然（不硬推）
- 技术术语是否有业务化解释或场景化表达

**传播友好：**
- 标题是否包含：产品 + 动作 + 价值
- 是否有可被单独传播的金句
- 结尾是否有低门槛行动召唤（"免费"、"两步"、"扫码"）

**专业可信：**
- 是否引用了真实数据或权威报告
- 是否承认了局限性或挑战（增强可信度）
- 量化结果是否具体可验证
- 是否避免了"大幅提升"等空洞表述

## 语言风格

**公式 = 技术感 + 口语化 + 节奏感**

| 技巧 | 示例 | 效果 |
|------|------|------|
| **对比句式** | "从'写代码'到'写命令'" | 强化价值感知 |
| **术语+解释** | "生成嵌入Ticket（票据）" | 专业但不晦涩 |
| **口语化表达** | "这个大家每天都在看数取数的BI工具" | 拉近距离 |
| **短句+分段** | 多用短段落、列表、加粗 | 阅读节奏感强 |
| **动词开头** | "无需编写复杂代码"、"只需输入" | 行动导向 |

### 开篇风格（必须采用以下之一，严禁空洞宏大式开场）

1. **痛点切入式**："传统OpenAPI调用需开发者手动处理HTTP签名、分页、排序...技术门槛高、流程繁琐且易出错。近日，Quick BI正式开放其命令行工具（CLI）..."
2. **场景还原式**："数据分析师小王每天的工作是这样的：早上从5个系统导出数据，用Excel做VLOOKUP匹配..."
3. **数据震撼式**："78%的企业数据分析师，80%的时间花在清洗数据而非分析。"
4. **热点借势式**："当AI开始接管数据分析的最后一公里，你的BI工具准备好了吗？"

### Stripe 写作原则（贯穿全文）

- **Rule of One**：一个读者、一个大想法、一个承诺、一个行动号召
- **Rule of You**：句子以"你"或动词开头，让读者觉得在对自己说话
- **精确胜过简洁**：不追求短小，追求精准。"Are you being succinct or are you being vague?"
- **"Even if"句式化解异议**："[价值主张] 即使 [常见反对意见]"

## 配图标注

在文章中用 `[配图建议：xxx]` 标注配图位置和类型：
- **架构图/流程图**：技术方案、系统设计处
- **操作截图序列**：产品功能演示处（从输入到结果的完整流程，关键步骤红框标注）
- **前后对比图**：效果展示处（左右/上下分栏，数据差异一目了然）
- **数据可视化**：性能指标、趋势分析处
- **演示GIF**：操作流程、交互体验处（15-30秒）
- 每个场景示例建议配 3-4 张截图
- 每 500-800 字至少一张配图

## 输出方式（双通道：对话框 + 钉钉文档）

文章完成后，**同时通过两个通道输出**：

### 通道一：对话框直接输出

在对话框中直接输出完整的文章 Markdown 内容，方便用户即时阅读、审阅和提出修改意见。

输出格式：
1. 先输出文章标题（作为一级标题）
2. 输出完整正文（严格按模板结构）
3. 包含配图位置标注（`[配图建议：xxx]`）
4. 包含结尾行动召唤

### 通道二：同步写入钉钉文档

在对话框输出文章的同时，**自动同步写入钉钉文档**。文档按文章类型归入对应子文件夹。

**固定根目录（无需询问用户）：**
- **根文件夹 ID**：`a9E05BDRVQRkezKGCyXpkaDYJ63zgkYA`
- **链接**：https://alidocs.dingtalk.com/i/nodes/a9E05BDRVQRkezKGCyXpkaDYJ63zgkYA

**文章类型与文件夹映射：**

| 文章类型 | 子文件夹名称 |
|---------|-------------|
| 产品功能发布文 | `产品功能发布` |
| 深度客户案例文 | `客户案例` |
| 方法论/趋势洞察文 | `方法论与趋势` |
| 技术架构解析文 | `技术架构` |
| 产品对比/选型指南文 | `选型指南` |

**发布流程：**

1. 使用 `mcp__钉钉文档__list_nodes`（`folderId`: `a9E05BDRVQRkezKGCyXpkaDYJ63zgkYA`）查找对应类型子文件夹
2. 文件夹不存在则用 `mcp__钉钉文档__create_folder` 创建
3. 使用 `mcp__钉钉文档__create_document` 在该子文件夹中创建文档（`name`: 文章标题，`markdown`: 完整正文——不含一级标题，钉文档的文档名即标题）
4. 创建成功后，**在对话框输出的文章末尾附上钉文档链接**，方便用户直接跳转

**注意**：钉文档 markdown 中的换行必须使用真实换行符，不能用字面量 `\n`。

### 内容修改与更新

当用户要求调整内容时，**两个通道同步更新**：

1. **对话框**：直接输出修改后的完整文章内容（或仅输出修改部分，视改动范围而定）
2. **钉钉文档**：使用 `mcp__钉钉文档__update_document` 同步更新：
   - `nodeId`：之前创建文档时返回的 nodeId
   - `markdown`：修改后的完整文章内容
   - `mode`：`overwrite`（全量替换）；如仅追加新章节可用 `append`
3. 如果上下文中找不到文档 nodeId，使用 `mcp__钉钉文档__search_documents` 按标题搜索定位
