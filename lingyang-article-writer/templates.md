# 五种文章类型详细模板

## 模板A：产品功能发布文

**适用场景**：新产品/新功能发布、开发者工具推广、技术能力开放

**标题公式**：
- [产品名] + [核心动作]，让[价值]迈入[时代/阶段]
- [产品名]正式发布[功能]：从"[旧方式]"到"[新方式]"
- [N]大能力全面开放：[产品名][功能]正式上线

**结构模板**：

```
# [产品名] + [核心动作]，让[价值]迈入[时代/阶段]

[现状痛点，写得够具体够痛]。[产品发布消息]。
[核心价值一句话，用"从X到Y"句式]。

[配图建议：核心概念图/产品全景图]
[可选：获取方式链接]

## [N]大核心能力，完整[继承/覆盖]

[能力全景描述 + 数量 + 覆盖范围]

[配图建议：能力架构图/功能地图]

## [N]大高频应用，一句话直达结果

### 示例一：[场景名称]，从"[旧方式]"到"[新方式]"

**痛点剖析**：[具体问题，要写得够痛——"跨天开发"、"额外开发定时任务"这种程度]

**用到的[产品]能力**：[能力列表]

**[产品]一句话方案**："[真实可用的命令/操作]"

**效果展示**：[前后对比效果]

[配图建议：3-4张截图，展示从输入到结果的完整流程]

### 示例二：[同上结构]
### 示例三：[同上结构]
### 示例四（可选）：[同上结构]

## 免费[使用/安装]，[N]步开启高效体验

### 第一步：[动作]
[具体操作说明]

### 第二步：[动作]
[具体操作说明]

[配案例命令/操作截图]

## 结语

[产品愿景 + 邀请反馈 + 扫码/链接入口]
```

**成功要素检查**：
- 痛点要写得够痛（"跨天开发"、"额外开发定时任务"这种级别）
- 命令/操作要真实可用（不要虚构）
- 效果要有对比（"从X到Y"）
- 每个示例配3-4张截图，展示从输入到结果的完整流程
- 结尾低门槛行动召唤（"免费"、"两步"、"扫码"）

**参考案例**：Quick BI CLI工具宣传文章——传统OpenAPI繁琐 → CLI开放 → "从写代码到写命令"，六大核心能力、134项功能，四大高频场景各配3-4张截图。

## 模板B：深度客户案例文

**适用场景**：标杆客户背书、行业解决方案推广、同行引流

**标题公式**：
- 【客户品牌】如何用【产品】实现【具体效果】？
- 从【痛点A】到【效果B】：【客户】的【产品】实践之路
- 【时间】内【效果】：【客户】用【产品】解决了【痛点】

**结构模板（五幕式）**：

```
# 从[痛点A]到[效果B]：[客户名]的[产品]实践之路

## 第一幕：[客户名]的数据挑战

[企业背景介绍：行业、规模、业务特点]

[具体业务困境描述：痛点要写成行业共性问题，引发潜客共鸣]

[配图建议：客户企业/行业场景图]

## 第二幕：传统方案为什么不够用

[尝试过的方案列举]
[每个方案的不足之处]
[建立"传统方案不够用"的共识]

## 第三幕：选择[产品]的关键考量

[为什么选这个产品——自然带出产品，不硬推]
[实施过程关键节点]

[配图建议：方案架构图/实施路径图]

## 第四幕：落地成效

| 维度 | 之前 | 之后 | 提升 |
|------|------|------|------|
| [指标1] | [旧数据] | [新数据] | [提升幅度] |
| [指标2] | [旧数据] | [新数据] | [提升幅度] |
| [指标3] | [旧数据] | [新数据] | [提升幅度] |
| [指标4] | [旧数据] | [新数据] | [提升幅度] |

[多维度量化结果，百分比数据展示]

[配图建议：前后对比数据图表]

## 第五幕：可复用的经验

1. [经验一]
2. [经验二]
3. [经验三]

> "[客户负责人金句证言]"
> —— [姓名]，[客户名][职位]
```

**参考风格选择**：
- **Quick BI客户案例风格**：侧重品牌势能（极氪、太古可口可乐等标杆企业）
- **帆软FineBI风格**：量化最细致（多维度百分比数据轰炸，南方航空案例）
- **火山引擎DataLeap风格**：工程师说服力最强（"4人管理3000任务"）

根据目标受众选择参考风格：面向业务决策者用Quick BI风格，面向数据团队用帆软风格，面向工程师用火山引擎风格。

## 模板C：方法论/趋势洞察文

**适用场景**：建立行业影响力、吸引决策层关注、内容引流

**标题公式**：
- 2026企业数据能力建设的[N]个关键趋势
- 数据治理的工程化实践：从人工治理到DataOps
- 数据资产化路线图：从成本中心到价值引擎
- [方法论名称]：从[A阶段]到[B阶段]

**结构模板**：

```
# [趋势判断/方法论名称]：从[A阶段]到[B阶段]

## 正在发生什么

[行业变化描述]
[用数据和权威报告支撑：Gartner、IDC等]

[配图建议：趋势数据图表]

## 为什么会这样

[根因分析，技术视角]
[根因分析，业务视角]
[技术+业务双重视角交叉论证]

## [N]步法/[N]阶段框架

[可复用的方法论/框架]
[每一步/每一阶段的具体说明]

[配图建议：方法论框架图/流程图]

## 谁已经这么做了

### 案例一：[企业名]
[简短描述实践过程和效果]

### 案例二：[企业名]
[简短描述实践过程和效果]

### 案例三（可选）：[企业名]
[简短描述实践过程和效果]

## 如何开始

[产品如何帮助落地这套方法论——自然引入，此部分占比不超过全文20%]
[低门槛行动召唤]
```

**Dataphin特别适用**：将产品能力上升为行业方法论（OneModel/OneData），让读者学习方法论的同时自然接受产品理念。这是"方法论包装型"平衡模式的典范。

## 模板D：技术架构解析文

**适用场景**：吸引技术决策者、建立技术壁垒认知、开发者社区传播

**标题公式**：
- 亿级数据秒查背后的技术架构：[产品][技术点]解析
- 从传统BI到AI增强分析：架构演进与实践
- [产品][技术模块]优化：我们如何做到[具体效果]

**结构模板**：

```
# [标题：技术挑战 + 产品 + 效果]

## 这个问题为什么难

[用业务场景引出技术挑战——让非技术读者也能理解难度]

[配图建议：问题场景示意图]

## 架构设计

[核心技术方案描述]
[简化版架构图，突出核心组件和数据流，不堆砌细节]

[配图建议：架构图（必须，放在文字之前降低理解门槛）]

## 关键技术实现

### 技术点一：[名称]

[深入浅出讲解，有类比有原理]

### 技术点二：[名称]

[深入浅出讲解，有类比有原理]

## 性能验证

| 指标 | 数据 |
|------|------|
| [性能指标1] | [具体数据] |
| [性能指标2] | [具体数据] |
| [性能指标3] | [具体数据] |

[配图建议：性能对比图表]

## 工程实践与踩坑

[真实踩过的坑和解决经验]
[承认局限性——这反而增强可信度]

## 总结

[技术总结 + 未来方向]
```

**写法要点**：每个功能点配具体性能指标（火山引擎模式），承认局限性反而增强可信度，架构图放在"怎么实现"之前降低理解门槛。

## 模板E：产品对比/选型指南文

**适用场景**：捕获评估阶段的买家、进入候选名单

**标题公式**：
- 2026年[品类]选型指南：[N]个维度帮你做决策
- [产品A] vs [产品B]：[N]个关键维度深度对比
- 企业[品类]怎么选？一份实用决策框架

**结构模板**：

```
# [标题]

## 为什么需要选型

[选型背景：行业趋势、企业痛点]
[核心考量维度介绍——客观中立的开场]

## 关键维度对比

| 维度 | [产品/方案A] | [产品/方案B] | [产品/方案C] |
|------|-------------|-------------|-------------|
| [维度1] | [评价] | [评价] | [评价] |
| [维度2] | [评价] | [评价] | [评价] |
| [维度3] | [评价] | [评价] | [评价] |
| [维度4] | [评价] | [评价] | [评价] |

[每个维度的详细分析说明]

[配图建议：评分矩阵/雷达图]

## 场景化建议

### 场景一：[企业类型/行业/规模]
[推荐方案及理由]

### 场景二：[企业类型/行业/规模]
[推荐方案及理由]

### 场景三：[企业类型/行业/规模]
[推荐方案及理由]

## 选型决策清单

读者可直接使用的决策检查表：

- [ ] [考量点1]
- [ ] [考量点2]
- [ ] [考量点3]
- [ ] [考量点4]
- [ ] [考量点5]

## 总结

[客观中立的总结 + 如需进一步了解的引导]
```

**写法要点**：开场必须客观中立，对比维度来自真实技术/业务需求，场景建议按行业、规模、技术栈分类，给读者一个可以直接拿来用的决策清单。
